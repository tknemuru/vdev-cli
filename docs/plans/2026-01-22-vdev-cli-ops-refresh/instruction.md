[Epic]
vdev-cli の ops.md を、新 vdev フロー時代に適合させつつ、フロー仕様との結合度を下げた
「CLI 運用ガイド」として再定義・刷新する。

本作業の目的は以下の2点である。
1. 旧 vdev フロー時代の前提・表現を排除すること
2. vdev-cli 側の ops.md が、将来のフロー変更（役割名・プロセス名・運用ポリシー変更）の影響を極力受けない構造にすること

[Non-Goals]
- vdev-flow の内容そのものを書き換えること
- 運用フローの正（SoT）を ops.md に内製すること
- 実行主体・役割名・レビュー体制・介入ルールを ops.md に定義すること

[Overall Policy]
- ops.md は「vdev-cli の運用ガイド」に限定する
- フロー仕様・役割・意思決定プロセスは、別途定義されたフロー仕様（SoT）に委譲する
- Step番号ベースの手順説明は行わない
- 状態（gate）駆動で「許可される操作」を説明する
- 現行フロー固有の名称（例: 特定の役割名・プロセス名）を本文に書かない

[Target File]
- vdev-cli リポジトリの ops.md

[Task 1]
概要：
新しい ops.md の目次構造を設計する（全面改稿前提）。

要件：
- ops.md 冒頭に「このドキュメントの位置づけ」を明示すること
- 「本ドキュメントが扱わないこと」を抽象表現で明記すること
  - 実行主体・責務分担
  - レビューや承認の意思決定プロセス
  - 例外時の介入・履歴管理・統制ポリシー
  - マージ・リリース等の上位判断
- SoT 参照方針を明示すること（CLI仕様 / フロー仕様）

完了条件：
- 目次レベルで、フロー仕様との責務分離が明確になっている

[Task 2]
概要：
「Daily Workflow」「Step 1〜N」型の構成を廃止する。

要件：
- Step番号に依存した章立てを削除する
- gate（status）を起点とした説明構成に置き換える
- 「公式フロー」「DONEまでの唯一手順」といった表現を使用しない

完了条件：
- ops.md 全体が、状態駆動（gate → allowed actions）の構造になっている

[Task 3]
概要：
状態（gate）別に、vdev-cli が許可する操作を整理して記述する。

要件：
- NEEDS_INSTRUCTION / NEEDS_PLAN / NEEDS_DESIGN_REVIEW /
  DESIGN_APPROVED / IMPLEMENTING / NEEDS_IMPL_REVIEW /
  DONE / REJECTED / BROKEN_STATE を対象とする
- 各状態について以下を記述する：
  - 状態の意味（CLI視点）
  - 許可される vdev コマンド
  - 代表的なエラーや注意点
- 「誰が行うか」は一切記述しない

完了条件：
- vdev gate の出力を見れば、次に打つべきコマンドが判断できる

[Task 4]
概要：
差戻し（NEEDS_CHANGES）時の CLI 的挙動を明確化する。

要件：
- NEEDS_CHANGES が返った場合の状態遷移を説明する
- CLI が提供する保証範囲と、保証しない範囲を区別する
- 差戻し回数・Attempts 管理には言及しない

完了条件：
- 差戻し時に「何を再実行すればよいか」が CLI 視点で明確

[Task 5]
概要：
CLI 固有の運用情報・注意点を整理して残す。

対象：
- BROKEN_STATE の代表例と回復不能性
- 手動編集禁止・hash 不整合の注意
- stdin / CRLF→LF の扱い
- vdev sync / vdev new の挙動
- .claude 資産の同期ポリシー

要件：
- Claude Code や特定実行主体に依存する説明をしない
- 「必須手順」と「補助的 Tips」を明確に区別する

完了条件：
- vdev-cli 利用者が ops.md だけで CLI 操作上の事故を避けられる

[Verify]
- ops.md 内に、特定の役割名・登場人物名・プロセス名が書かれていないこと
- フロー仕様変更を想定しても、ops.md の内容が破綻しない構造になっていること
- vdev-spec.md の状態機械・コマンド仕様と矛盾がないこと

[Rollback]
- 現行 ops.md をそのまま保持し、変更を破棄する
